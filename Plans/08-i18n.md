# 08 — Internationalization (PL + EN)

## 1. Context

All customer-facing copy is English-only, but it is already centralized in `instant-quote/src/lib/strings.ts` (a typed `as const` object covering hero, quote UI, DFM labels, order dialog, footer) — which makes this topic far cheaper than typical i18n retrofits. Formatting helpers exist in `src/lib/format.ts` (`formatPln` etc.) but locale-awareness is unverified. There is no locale routing, no language switcher, no translated DFM warnings or error messages, and emails (plan 06) and legal pages (plan 09) need PL+EN from day one. Launch requires: a Polish user completes quote → payment → emails entirely in Polish; an English user entirely in English.

The Go pivot adds one new surface: API responses. The backend returns DFM flags, validation errors, and status labels that the UI displays.

## 2. Decisions applied

**Pinned (DECISIONS.md):** typed two-locale dictionary extending `strings.ts` — no i18next; `/pl` (default) + `/en` path prefixes with `hreflang`. Prices stay PLN in both locales.

**Topic-local decisions:**

- **Go returns machine codes, the frontend dictionary owns all human text.** DFM flags, API validation errors, and order statuses come back as stable codes (`dfm.too_large`, `error.invalid_nip`, `status.in_production`) with structured params (`{maxMm: 340}`); the typed dictionary maps code → localized string. *Rationale:* one place owns copy, translations never require a backend deploy, and the OpenAPI schema can enumerate the codes so exhaustiveness is compile-checked on the frontend. *Exception:* emails and invoices are rendered backend-side (plans 06/05) — their locale is persisted per order/user and their copy ships in the pre-rendered template artifacts (which the shared dictionary feeds at build time).
- **Dictionary shape:** `strings.ts` becomes `src/lib/i18n/pl.ts` + `en.ts`, both `satisfies Dictionary` against a single inferred type — adding a key to one locale without the other is a type error. A `useStrings()` hook resolves the active locale; no runtime framework.
- **Default locale PL** (`.pl` market), EN secondary; root `/` redirects by `Accept-Language` with a persisted override (cookie) from the switcher.
- **Admin (`/admin/*`) is exempt** — EN-only operator tool (confirmed with plan 07); excluded from the no-hardcoded-strings check.

## 3. Implementation phases

### Phase A — Dictionary split + typed plumbing

- Create `src/lib/i18n/` (`types.ts` inferring `Dictionary` from `pl.ts`; `en.ts` `satisfies Dictionary`; `index.ts` exporting `useStrings()`/`getStrings(locale)`). Move `strings.ts` content to `pl.ts`-shaped keys with the current English text as the `en.ts` seed; translate PL.
- Interpolation helper for parameterized strings (`{maxMm}`), typed per key.
- **Verify:** typecheck fails when a key is removed from one locale only (add a deliberate red test then fix); all existing components compile against `useStrings()`.

### Phase B — Locale routing, detection, switcher, hreflang

- Route tree gains a `$locale` segment (`/pl/...`, `/en/...`) validating against `['pl','en']`; `/` → redirect per cookie → `Accept-Language` → `pl`. Internal links go through a locale-aware `Link` wrapper.
- Language switcher in `SiteHeader` (both variants), swapping the prefix in place and setting the cookie.
- `hreflang` alternates + `x-default` in the document head per route (feeds plan 13's SEO).
- **Verify:** `/pl` and `/en` render the full quote flow in the right language; switcher preserves the current page; `curl -H 'Accept-Language: en'` on `/` redirects to `/en`.

### Phase C — Translate everything + API code mapping

- Sweep every component for hardcoded strings → dictionary (DfmBadges, error toasts, OrderDialog validation, PriceBreakTable headers, HowWePriceDialog body, dropzone copy…). Enforcement: an oxlint rule or a grep-based CI check (`scripts/check-strings.ts`) failing on JSX literals outside `src/lib/i18n/` (admin routes exempt).
- **Backend codes:** audit `backend/api/openapi.yaml` responses — DFM flags and error envelopes gain `code` + `params`; Go handlers stop emitting prose. Frontend maps codes via the dictionary (`dfm.*`, `error.*`, `status.*` namespaces); unknown code falls back to a generic localized message + Sentry breadcrumb (plan 11).
- Zod-on-the-client validation messages (email, NIP) localized via the dictionary.
- **Verify:** the string check passes; forcing each DFM flag (oversize part, thin walls, hull fallback) shows translated badges in both locales; an API validation error displays localized.

### Phase D — Locale-aware formatting + persistence plumbing

- `format.ts`: number/date formatting via `Intl.NumberFormat`/`DateTimeFormat` with the active locale (PLN symbol placement differs: `12,34 zł` vs `PLN 12.34` — pick `zł` suffix for both, PL decimal comma vs EN point). Ship-date labels: the backend's canonical labels (`internal/leadtime`) are EN-shaped (`"Thu 16 Jul"`) — return the date ISO alongside and format client-side per locale instead of displaying the label raw.
- Persist locale where downstream plans need it: order creation sends the active locale (plan 05's `orders.locale`); registration sends it (plan 04 — flagged there).
- **Verify:** prices show `1 234,56 zł` on `/pl` and `1,234.56 zł` on `/en`; ship dates render localized weekday/month; a `/pl` order stores `locale='pl'` and its emails arrive in Polish (plan 06's checklist covers the send side).

## 4. Dependencies

- **Hard:** none to start — begin any time; **earlier is cheaper** (every new component written after this lands uses the dictionary from day one).
- **Feeds:** 06 (email dictionary namespace + persisted locale), 09 (legal pages in both languages), 13 (hreflang/sitemap), 05 (`orders.locale` — column already in its migration), 04 (user locale — open flag there).
- **Blocks:** launch, 09.

## 5. Verification

- [ ] Full PL journey: `/pl` → upload → quote → checkout → confirmation, every visible string Polish; EN likewise.
- [ ] No hardcoded string outside the dictionary: CI check green (admin exempt).
- [ ] Dictionary exhaustiveness is compile-enforced (removing a key from one locale fails typecheck).
- [ ] All DFM flags + API error codes render translated in both locales.
- [ ] Number/date formatting locale-correct; PLN in both.
- [ ] hreflang alternates present and reciprocal on every public route.

## 6. Risks & open questions

- **Translation quality.** Machine-drafted Polish must be reviewed by a native speaker before launch (the shop's home market reads it first) — schedule alongside plan 09's lawyer pass.
- **Backend prose audit.** The existing OpenAPI error/DFM shapes were written pre-i18n; converting to codes is a breaking change to the contract — do it before the frontend hardcodes English prose from responses. `gen-check` + one coordinated PR.
- **Route-tree churn.** Adding `$locale` re-prefixes every public route; do it before plans 04/05/09 add more routes, or their PRs collide.
- **Ship-date label duality.** The Go engine's canonical labels stay (tests pin them); the UI just prefers the ISO date. Don't localize labels backend-side.
- **Open:** URL slugs (`/pl/kontakt` vs `/en/contact`) — recommend localized slugs only for the handful of content pages (13/09), keeping app routes identical across locales.
